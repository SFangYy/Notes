从 Notion 迁移到 Obsidian 后，一直在尝试将以前构建的 P.A.R.A 系统也迁移到 Obsidian 中，期间也是折腾了不少插件、写了不少代码，在踩了不少坑、迭代了好几版后，终于是搭建出来了一个简单、高效的行动管理系统。

这个系统只依赖 2 个插件、几近 0 代码，却帮我高效管理了多个领域，累积执行项目数超 1000 个，整体完成率达到了惊人的 90%。

不仅帮我完成了生活、工作的有序管理，现在更是我的长期主义执行引擎，为我提供源源不断的执行动力。

## 0、源于 P.A.R.A

系统基于 P.A.R.A 构建，这是因为 P.A.R.A 的分类原则和管理目标都是赋能行动，与我的目标高度契合。

> 什么是 P.A.R.A： https://fortelabs.com/blog/para/

我使用标签将文件分为 2 大类，分别是领域（area）和项目（project）

- **项目指的是拥有终止状态和截止日期的事物**，比如明天下班前完成 xxx 合同初稿拟定
- **领域是个人需要长期投入和持续维护的事物**，比如工作、生活等，每个项目必然会关联一个领域。
- 我在项目和领域上添加了一个归档属性，该属性是一个复选框，如果勾选的话说明该文件已经归档了

我舍弃了原 P.A.R.A 中的资源概念，单独建了一个「兴趣探索」的领域来替代，一些感兴趣的事项我会作为项目关联到该领域。

因为我觉得仅仅是感兴趣却没有作为行动去进一步探索的话，资源只会沦为一个吃灰的收集箱。

![[20251006101038.png|领域和项目]]

## 1、我的目录结构

我折腾过很多版本的目录结构，从粗放的管理到细致的管理都有尝试，比如

- 一个项目一个文件夹，然后与项目关联的文件全放那个文件夹，好处是可以将项目直接打包给其他人，缺点是维护起来心累，比如项目文件改名了还得同步去该文件夹的名字（尽管有插件能做到）
- 所有文件全放一个文件夹，好处是存放简单，但要是文件名或属性设计不好，找个文件得找半天

经过多次迭代后，总算是找到了一个适合自己的平衡点。（本文主要重点关注是 area 和 project 两个目录）

```markdown
area/  ---- 领域文件
	工作.md
	生活.md
dashboard/ ---- 基于 componnets 构建的仪表盘
base/ ---- 知识类笔记，会不断迭代修改的笔记
ouptut/  ---- 输出文件
project/ ---- 项目文件
	2250901 xxx 项目合同初稿.md
assets/
	weread/ ---- weread 插件同步的读书笔记
	excalidraw/ ---- excalidraw 创建的文件
	attachments/ ---- 附件
```

- area 目录存放了所有类型为 area 的文件
- project 目录存放了所有类型为 project 的文件

area 目录还好，因为领域的文件数并不会太多；但是 project 目录下会有成百上千个文件，直接通过目录来管理和定位项目文件会很繁琐耗时。

所以我使用 componnets 插件创建了一个看板视图，该视图可以筛选出符合条件的文件，并且按项目状态分组展示，打开该看板我就能知道自己当下该做什么，已经完了什么。

![[20251004181039.png|项目看板]]

## 2、以文件为基本管理单位

我的每一个领域和项目都是一个 markdown 文件，这是因为
- 在 Obsidian 中 markdown 有标准的属性规范和结构，每个 Obsidian 用户都遵循这套规范
- Obsidian 的基本索引单位是文件，而非文本块，所以基于文件做统计会更高效和便捷
- Obsidian 提供了非常便捷的文件管理操作，比如重命名时自动更新关联引用
- 相对于以 task 的形式来记录，投入的系统构建成本会低很多

当然，我曾经也担忧过过多的文件是否会给管理带来困难，是否会给性能带来压力，但是到现在我发现这些担忧其实都无足轻重
- 一是当下大多数用户的设备性能可以轻松的处理数十万文件，大部分用户文件是 1000 以下，能创建出超 10w 个文件的用户已是凤毛麟角。所以不妨等先达到 5w 笔记再去考虑 10w 的问题
- 二是通过合理的文件名+属性设计，配合 components 的数据视图（或 Obsidian 最新的  bases）可以帮我直接汇总和管理我想要的数据，而不用再去文件夹里面倒腾

## 3、属性设计

领域和项目文件有着不同的属性设计，这两者的区分也是同过 tags 属性的值来做区分的

- tags 属性的值为 `area` 表示该文件是领域文件
- tags 属性的值为 `project` 表示该文件是项目文件

这样设计的好处是通过修改 tags 的属性值即可切换一个文件的类型；通过 tags 筛选也可以快捷的汇总某个类型的数据。

我的领域笔记属性展示

![[20251006191031.png]]
项目笔记的属性会复杂一点，因为还涉及到状态管理、数据统计等，有 5 个属性，分别是

- **项目状态**： 标记当前项目的执行状态，比如执行中、已完成等
- **创建时间**： 项目创建的时间，在项目创建的时候就自动记录上了
- **完成时间**： 项目完成的时间，只有在项目完成时或取消时才会有记录
- **领域**：通过双链语法关联到一个领域文件，比如 `[[工作]]`（领域表示需要长期投入和持续维护的事物，比如工作、生活等）
- **tags**： 填写固定值 `project`， 用于标记该文件是项目类型

### 3.1、项目状态

状态属性是整个管理流程的核心，有 4 种值，分别是

- TODO （待办）
- DOING（正在做的）
- DONE（已完成）
- CANCELED（已取消）

![[20251003221011.png|状态流程图]]

这里我重点说一下 TODO 和 DOING 两个状态的设计。

我会把想做的事情都作为 TODO 项目记录下来，记录时不会做复杂的决策，主要的决策点有两个
> 快速记录我采用 Form flow 插件来构建的表单，可以帮助我自动的填写创建时间等属性
> https://form.cc1234.cc

- 这件事情是什么？通过标题简洁的说明
- 这件事情归属于什么领域？我需要确保自己的长期努力都投入在了期望的领域中 

在我这里 **TODO 类项目不是债务，而是领域投资**，只不过这些投资项只有执行完成以后才能拿到最后的收益。所以我会定期的通过仪表盘去回顾 TODO 项目列表，将其转换到 DOING 中去。（在我不知道做什么的时候我也会来看看 TODO 项目列表）

再说说 **DOING， 这是当前系统中最重要的事项，它除了表示当前项目正在进行中外，还表示当前项目就是最高优先级的项目**。（正因如此我的系统中没有优先级这个属性）

为了确保自己始终专注在**项目的完成上**， 我会严格控制每个领域的 DOING 状态项目数在 7 个以下，避免过载导致的失控，一旦 DOING 数超过 7 个就需要先停下来把低优先级的事项拖到 TODO 中去。

### 3.2、 项目领域

为每一个项目关联一个领域，可以强制让我坚持在一个预定的跑道上去做执行。

虽然在创建项目的时候强制要求选择领域会给创建这个动作带来“中断”感，但相对于行为“发散”带来的虚假忙碌，我倒觉得这点中断损失可以忽略不计。

文件中的 area 属性值采用的是双链格式。这样的好处是能直接点击链接跳转到领域文件中去

### 3.3、 创建时间

创建时间是最基础的属性，主要作用还是用于做数据统计。

**那为什么不用文件自带的文件创建时间呢？**

因为自带的在某些场景下值会变。

比如多端同步场景下，在电脑端的创建时间是正确的 2025-01-01，但是在手机端同步时需要从云端下载文件，这个时候可能文件创建时间就变成了下载文件那一刻，比如 2025-10-01 了，会给数据统计带来麻烦。

### 3.4、 完成时间

这个是在项目状态被更新为 DONE 的时候的时间，主要作用是用于统计项目耗时、或者某段时间内完成的项目数。

有时候我也会把这个时间作为「计划完成时间」来使用（由于目前还没有统计需求，所以计划完成时间我也没作为单独的属性来维护）

我在 components 的看板中添加了一个「完成」按钮，点击即可一键将项目状态更新为 DONE， 将更新时间更新为当前

###  3.5、 扩展原则

对于一些成熟的领域，我会按需来扩展属性从而服务一些统计需求。

扩展遵循奥卡姆剃刀原则，即**如果你当下不添加这个属性对你的系统也没有任何功能影响，那就不用去添加。**

谨记不要为臆想的场景去做设计，因为每增加的一个属性都会带来额外的维护成本，为系统管理增加复杂度。

举个例子，我在每次 components 版本发布的时候需要将更新日志按类型区分，从而生成一个发布日志给到用户，让用户知道增加了什么功能、修复了什么 bug。由于这是已经实打实存在的需求，所以我为 components 领域的项目增加了一个 `项目类型` 属性，从而便于统计生成发布日志，这就是根据实际需求去添加属性。

## 4、效率实践

### 4.1、快速创建项目文件

在创建项目的时候需要添加很多属性，文件名也需要添加一个日期前缀，如果通过手动来执行这些操作会很繁琐，所以我使用了 Form flow 来将这些操作自动化

> 插件尚未上架，请通过官网获取 https://form.cc1234.cc

![[20251006201011.png]]


### 4.2、子任务拆分

对于复杂的项目，我会在项目文件中通过 task 的形式来拆解子任务，但还是尽量保持项目简单，这样可降低管理复杂度、并提高执行效率和成功率

![[20251004201028.png]]

配合 components 的任务清单功能，可以直接在视图上显示笔记内的子任务项

![[20251006191003.png]]

如果子任务还需要继续拆分的话，可以将子任务作为一个独立的项目文件创建出去，不过这种情况我倒是很少遇到

### 4.2、自动化项目完成时的步骤

完成项目时需要更新项目状态和项目完成时间，前面提到的在 components 视图中通过按钮来一键执行适用于当前在视图页面的场景。

但有时候我们会在项目文件中，这个时候我会使用 Form flow 来实现一键完成的动作

![[20251006201021.png]]



## 5、 放在打开就能看到的地方

我把各个领域要做的事情放在了主页上，这样我每天打开 Obsidian 就能立刻知道今天该做什么，从哪儿开始。

这样一个简单的动作，可以极大的提高项目的执行完成率
